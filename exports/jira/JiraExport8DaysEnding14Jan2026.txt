h2. Old description with discussion of the possible options

Rancher stores user passwords as a bcrypt hash in a custom user local resource. 

SUSE Observability stores user passwords as bcrypt hash in the config file. Rancher however uses a temporary bootstrap password, after which a real user password must be provided and is stored into the admin User ([users.management.cattle.io|http://users.management.cattle.io]). A similar approach would be user friendly but also a lot of work to build (requiring a UI for providing the new password).

The main problem is that providing the real password via Helm values will put it in plain text in the Helm release secret, making it less secure. Any other way to avoid this is also acceptable. Storing it in a secret that we automatically reload when it changes or does not exist is also fine.

Another way is to generate the secret only when the secret doesn’t exist yet (but this does come with a bunch of caveats and issues). See [https://stackoverflow.com/questions/56170052/how-not-to-overwrite-randomly-generated-secrets-in-helm-templates|https://stackoverflow.com/questions/56170052/how-not-to-overwrite-randomly-generated-secrets-in-helm-templates|smart-link] for a solution (the one by {{akop}}) that only won’t work when using {{helm template}} (most importantly for tools like argocd) and {{helm update --dry-run}} (it will think there is no secret unless specifying {{--dry-run=server}}). This can/should still be combined with the ability to provide the bcrypted password explicitly for a pre-defined password.

The solution we’re going for without this ticket is to have the user provide a bcrypt hash as a variable that we allow them to create using one of these (tbd):

# very simple helm chart that just generates the password hash (this is convenient because they already have helm, but annoying because they still need a second, tiny, chart)
# add a {{password}} command to our {{sts}} cli and include a download link in the docs for the CLI.
# Use an existing tool like {{bcrypt-tool}}: [https://github.com/shoenig/bcrypt-tool|https://github.com/shoenig/bcrypt-tool|smart-link] 

Note that we do have a bootstrap service token that is stored in plain text, but by default it expires after 24h.

An alternative option is to solely rely on the “security” of the Kubernetes secrets (the fact that only admins can access the secrets for SUSE Observability) and allow the password as plain text value in the values.yaml (we can still bcrypt it in the helm chart for usage in the config file). This is however considered bad-practice and a step backward from the current situation. It may also come back later as something that needs to be fixed.
